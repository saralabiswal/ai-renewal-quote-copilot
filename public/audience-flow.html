Static IA concept

Regroup the app by user intent, not by feature inventory.

The app has strong pieces already: renewal operations, policy transparency, AI architecture, and developer validation. The cleaner story is to present those pieces as three role-based workspaces with a shared decision trace running through all of them.

Three User Types

Each audience should have a clear entry point, a short set of primary tasks, and a proof surface that tells them whether the system is doing the right thing.

Business User

Revenue leaders, account teams, deal desk, and reviewers who need to prioritize renewals, inspect recommendations, apply quote actions, and approve outcomes.

Architecture User

AI governance, enterprise architects, and technical evaluators who need to understand policy boundaries, model behavior, LLM controls, and traceability.

Developer User

Engineers who need to run the project, reset data, validate contracts, inspect implementation surfaces, and extend the workflow without breaking the demo story.

Recommended First-Run Flows

Each audience gets a four-step journey. They can overlap later, but the starting sequence should feel intentional.

1

Business demo

Renewal Subscriptions -> Baseline Quote Review -> Scenario Quote Review -> optional Scenario Quote Generation Trace.

2

Architecture review

Decisioning Setup -> AI Architecture -> Policy Playbook -> Scenario Quote Generation Trace.

3

Developer setup

README -> clean DB reset -> build/test -> inspect workflow APIs.

4

Shared handoff

All roads end in the same artifact: a quote decision backed by evidence.

Implementation Recommendation

This can be added without rebuilding the whole product. Treat it as an information architecture layer over the existing routes.

1. Rename the top-level areas

Use Business Workspace, Architecture Console, and Developer Workbench as the main sidebar groups. Keep existing route names inside those groups.

2. Add an audience switch on the dashboard

Let the user pick Business, Architecture, or Developer. Change the dashboard cards and next actions based on that selection.

3. Make Decision Trace the shared proof object

Business sees explanation and approval evidence. Architects see rule, ML, LLM, and guardrail evidence. Developers see payloads and API contracts.

4. Split Decisioning Setup into readiness and internals

Show a simple operating posture first. Put provider values, registry detail, and guarded mode internals in architecture/developer views.

Best next product move: turn this static concept into a new in-app "Flow Map" page, then update the sidebar grouping and dashboard entry cards. That gives every user a self-explanatory path while preserving the current workflow implementation.
